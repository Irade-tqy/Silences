# Baseline v1 — 实际跑分记录

> 日期：2026-07-02  
> 模型：`deepseek-v4-flash`  
> 测试项目：dailyPlanner — Next.js 日历+番茄钟  
> 操作人：@silences

---

## 1. 方法说明

Baseline 不是实验室测试，是**真人实际操作**。操作人对 dailyPlanner 代码库熟悉，了解 bug 现象，但不预知具体修复方式。

两轮测试的操作顺序：

1. **CC 先跑**：操作人第一次做这个 benchmark，不知道 bug 根因，不知道框架会在哪里犯错
2. **CodeWhale 后跑**：操作人已经知道 CC 踩过哪些坑，human feedback 更精准，反馈更高效

**关键**：CodeWhale 占尽了 human 经验优势，但依然更贵。这才是对比的意义。

## 3. Claude Code (CC)

```
Cache Hit:    8,752,896  (94.8%)
Cache Miss:     419,649  (4.5%)
Output:          65,693  (0.7%)

Total Cost:      ￥0.726
```

**关键数据**：
- 操作人盲跑，多轮迭代修好
- 缓存命中率 94.8%——很高，说明 CC 的上下文管理有效
- 输出 token 65k——因为多轮试错，agent 输出了更多代码解释
- **￥0.726 是标杆**：一个成熟框架在盲跑条件下的成本

## 4. CodeWhale

```
Cache Hit:    9,351,936  (93.7%)
Cache Miss:     625,577  (6.3%)
Output:          46,653  (0.5%)
─────────────────────────────────
Total Cost:      ￥0.906
```

**关键数据**：
- 操作人有经验，一轮精准反馈修好（vs CC 多轮）
- 缓存命中率 93.7%——比 CC 低 1.1%，绝对值多 206k 未命中
- Cache Miss 比 CC **多 49%**——这是 CodeWhale 更贵的根因
  - 每轮传递了更多无法缓存的上下文
  - 上下文管理中存在冗余 token 传输
- 输出 token 46k——比 CC 少，符合「一轮搞定」的预期

---

## 5. 对比总结

| 指标 | CC | CodeWhale | 差距 |
|------|-----|-----------|------|
| Cache Hit | 8,752,896 | 9,351,936 | +6.8% |
| Cache Miss | 419,649 | 625,577 | **+49.1%** |
| Output | 65,693 | 46,653 | -29.0% |
| 操作人经验 | 首次（盲跑） | 第二次（有经验） | — |
| **总价** | **￥0.726** | **￥0.906** | **+24.8%** |

### 核心发现

**CodeWhale 在占尽 human 经验优势的条件下，仍然比 CC 贵 25%。**

如果两个都盲跑，差距只会更大。CodeWhale 的 49% 额外 cache miss 暴露了其上下文管理机制的缺陷：每轮有大量冗余 token 无法命中缓存。

---

## 6. Silences 的目标

```
Silences Target: ≤￥0.50  盲跑一轮搞定

约束条件：
  缓存命中率 ≥ 96%         (CC 是 94.8%，目标是压住 cache miss)
  输出 token ≤ 50k         (一轮修复预期)
  Human 盲跑               (和 CC 同条件对比)
```

### 关键杠杆点

| 优化点 | 影响 | 说明 |
|--------|------|------|
| 上下文去重 | 降 miss | 每轮只传变动的部分，不变的进缓存 |
| 前缀缓存策略 | 升 hit | 确保 system prompt + 文件内容稳定，最大化缓存复用 |
| 控制 tool 结果 | 降 miss | 允许精细选择哪些结果传给模型（但保留上下文） |
| 无框架 overhead | 降 total | 不像 CodeWhale 有额外封装成本 |
