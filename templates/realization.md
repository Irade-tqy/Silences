# 再给实现

当你 rollback 了目标环节，根据是否从零开始，开始并 checkpoint 后续环节。

**禁止在这里修改代码**

假如是从零：

**锚定基座**：

最主要的：什么语言、什么平台、哪个包管理器？等等。

用户在哪？如果是 Github Pages，那么需要静态网站；如果是二进制，那可能要考虑 Electron 的性能问题……

用户选择了之后，让用户去装需要的软件（如果还没有）。禁止自己装软件。

**艺术风格**：

在这件事上，你必须完全听用户的。

不要自己想一个艺术风格，参考已有的软件，给用户提供两个选项。用户可以自己去找。

然后让用户去总结那个网站的艺术风格反馈给你。

告诉用户你需要哪几个板块、什么格式。比如配色、字体、线条、布局。

**开发方式**：

敏捷开发？逐步开发？混合？

假设你的工作能力是正常一个全栈工程师的 5 倍左右，来制定一个简单的开发计划（不要事无巨细，就说这个阶段我要做什么功能，和目标对应），以工作日为单位。

在和用户讨论通过后写入会话的 PLAN.md。

假如不是从零：

**现有的代码**：

现在的实现是什么？

多思考一层：它是解决什么问题的？我改了会不会回归？

**要改的部分**：

我该怎么修改当前代码？

多思考一层：这是最简洁的吗？有没有不必要的检查和废代码？

再思考一层：哪里需要 try-catch 和 log？

**计划列表**：

提供一个拆到模块级别的列表。

在和用户讨论通过后 rollback，将其写入会话的 CONTEXT.md。
