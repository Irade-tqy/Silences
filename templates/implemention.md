### 拆任务并执行

先（init 仓库并）commit。

现在你有了一个 PLAN.md 或是一个计划列表，按计划 add & start & end task 并执行。

子任务可以继续拆分子任务：比如执行任务探索，发现要探索前端、后端、数据库逻辑。那么就可以分别 add task 前端、后端、数据库，再 add task 继续探索，然后 end task 探索。

> **end_task 机制**
> 
> 当你 end 一个 task 时，系统会自动请求你更新 CONTEXT.md，你只要照做，禁止忽略这条消息。
> 
> 然后系统会压缩上下文，你将不再记得之前任务的细节，只会获得总结和 CONTEXT.md。

**约束**：

- 最终任务原子化：每个任务只做一件事。例如不写“修复登录并加缓存”。如果是两条，继续拆分子任务
- 包含测试：一般“编译与验证”作为最后一个任务完成
- 实时更新：禁止不 end_task 就执行下一个任务

如果在执行过程中发现了无关的 bug 或者其他不符计划的地方，建议停下来、提出来，但禁止随手修掉。当然如启动脚本写错、语法问题、lint warning 等工程小问题可以顺便优化，但必须如实记录在会话的 LOG.md（不存在则创建）。

当尝试同一个任务（如 debug），从不同方面思考了大于等于三次，且毫无进展时，必须获取更多的上下文：停止挖得更深，视任务类型添加调试或请求用户的其他帮助。禁止死磕。

底线：禁止把问题还给用户，告诉用户你的发现和思路，询问用户的思路，或者给出具体的操作步骤让用户去做。
